# ADR-0001: Operational Readiness Before Production Persistence

## Status

Accepted

## Context

Phase 2 stabilized the BBIOS OS architecture around FastAPI, request handlers, application services, repository protocols, JSON repository implementations, and JSON persistence.

The next major technical direction is production persistence, but the current service still needs stronger operational foundations before database migration begins. The repository already contains process-local structured observability, settings, health behavior, and handler-level request tracing. These capabilities should be strengthened before adding PostgreSQL complexity.

## Decision

BBIOS OS will complete Phase 3.1 Operational Foundation before Phase 3.2 Production Persistence.

Phase 3.1 will prioritize correlation IDs, request tracing, centralized FastAPI exception handling, health and readiness semantics, process-local metrics, and operational documentation.

PostgreSQL, SQLAlchemy, Alembic, runtime persistence switching, and JSON replacement remain deferred until Phase 3.1 is complete and approved.

## Alternatives Considered

Starting PostgreSQL immediately was rejected because persistence migration would increase operational risk before request tracing, readiness, and error handling are mature.

Adding deployment, Docker, or CI/CD first was rejected because runtime observability should exist before deployment hardening.

Leaving observability as-is was rejected because handler-level tracing exists, but the canonical FastAPI boundary needs consistent operational infrastructure.

## Consequences

Operational maturity becomes the next engineering gate before database work.

Phase 3.1 work must remain compatibility-first and additive. Existing routes, response envelopes, status codes, service contracts, repository contracts, and JSON persistence formats must remain unchanged.

Future PostgreSQL work can use the operational foundation for readiness checks, tracing, metrics, and safer rollback validation.
